Executive Summary

Sentinel is an open-source autonomous financial research agent built by Rajkaran Yadav that has attracted significant community interest — amassing over 17,400 GitHub stars and 2,100 forks since its January 2026 launch. Built on TypeScript with the Bun runtime, Sentinel demonstrates a compelling proof-of-concept for domain-specific agentic AI: it decomposes complex financial queries into structured task plans, executes them using live market data tools, self-validates findings, and iterates toward confidence before surfacing results.

However, its current architecture is constrained to a single-user CLI environment, a narrow financial data toolset, and a stateless, single-session memory model. It lacks the resilience, observability, multi-tenancy, and extensibility required to serve institutional users or to compete with emerging enterprise-grade AI research platforms.

Document Objective

This Business Requirements Document (BRD) proposes a comprehensive set of enhancements to evolve Sentinel into a top-tier, industry-grade AI agent platform — one that retains its domain-specific intelligence and research quality while adding the architectural depth, extensibility, and enterprise readiness required for professional deployment at scale.

The proposed enhancements span eight capability domains: agent orchestration, persistent memory, tool ecosystem expansion, advanced reasoning, multi-agent collaboration, adaptive learning, real-time streaming, and security/compliance. Together, they position Sentinel not merely as a research tool but as a full-featured agentic intelligence platform for financial services.

Current State Analysis

2.1 Architecture Overview

Sentinel's current implementation is a TypeScript monorepo organized into five primary modules. The agent loop (src/agent/agent.ts) is the execution core, running an iterative tool-calling cycle with a configurable maximum iteration count (default: 10). The tool registry (src/tools/registry.ts) conditionally includes tools based on available environment variables. A scratchpad (src/agent/scratchpad.ts) provides the sole source of truth for tool results within a single query session. Skills (src/skills/) are SKILL.md-based extensible workflows — most notably a DCF valuation skill — that the LLM invokes via a dedicated skill tool.

2.3 Identified Gaps

Three structural gaps limit Sentinel's ceiling as a professional platform:

Critical Gap — Stateless Memory

Every query begins with a blank slate. There is no mechanism to carry forward prior research findings, established facts about covered companies, or user preferences between sessions. This fundamentally limits the depth of ongoing coverage and forces redundant retrieval on every query.

Critical Gap — Single-Agent Bottleneck

All reasoning, tool selection, execution, and validation flow through one LLM call sequence. Complex research tasks that could benefit from parallelism — comparing five companies, cross-referencing macro and micro data simultaneously — are serialized unnecessarily, increasing both latency and token cost.

Critical Gap — No Deployment Surface

The system has no REST or WebSocket API, no authentication layer, and no multi-tenancy model. It cannot be embedded into existing enterprise workflows, BI tools, or institutional research platforms without fundamental re-architecture.

Strategic Vision

"Sentinel should become the intelligent research backbone that institutional analysts reach for first — not because it is automated, but because it thinks like the best analyst on the team and never forgets a piece of prior work."

The strategic ambition is to evolve Sentinel from a single-user CLI research assistant into a multi-tenant, agent-native intelligence platform purpose-built for financial services. This means retaining the qualities that differentiate it — tight financial domain expertise, transparent reasoning chains, and honest self-validation — while expanding the platform along three axes:

Axis 1 — Depth of Intelligence

Richer reasoning through multi-agent specialization, persistent institutional memory, and continuous self-improvement via online learning from user feedback. The agent should get demonstrably smarter at covering a specific company or sector over time.

Axis 2 — Breadth of Deployment

A REST/WebSocket API layer, web UI, and native SDK enabling embedding into Bloomberg Terminal workflows, internal data portals, Slack bots, and mobile research apps. Any surface where a human analyst works should be able to call Sentinel.

Axis 3 — Trust and Accountability

Financial decisions carry real consequences. Every Sentinel output must be auditable: full provenance chains linking each analytical claim back to its source data, timestamps, and tool call parameters. Institutional compliance requirements demand no less.

Core System Design Enhancements

4.1 Agent Orchestration Engine

The current linear agent loop must be replaced with a structured, DAG-based orchestration engine that supports dynamic task graphs, parallel execution branches, and inter-agent messaging.

REQ-ORCH-01 Hierarchical Planner–Executor Architecture

Introduce a dedicated Planner Agent responsible for decomposing queries into a formal task DAG, distinct from the Executor Agents that carry out individual research tasks. The planner should output a structured JSON plan with task nodes, dependencies, estimated token budgets, and parallelism hints.

  • Planner produces a typed ResearchPlan schema with explicit dependency edges between tasks
  • Executor pool instantiates parallel agents for dependency-free task branches
  • Plan is versioned and stored for post-hoc audit and replay
  • Users can inspect, edit, and approve the plan before execution begins (configurable)
  • Dynamic plan revision when tool calls return unexpected results or data gaps
REQ-ORCH-02 Adaptive Loop Control with Cost Budgeting

Replace blunt max_steps integer limits with a multi-dimensional budget controller that monitors token consumption, wall-clock time, API cost, and confidence convergence simultaneously.

  • Per-query configurable budget: { maxTokens, maxCostUSD, maxSeconds, minConfidence }
  • Real-time budget dashboard visible in the CLI and web UI during execution
  • Graceful degradation: agent surfaces partial findings with explicit confidence flags when budget is exhausted
  • Loop detection upgraded from simple counter to semantic similarity hashing of reasoning states
  • User-configurable early-stop thresholds: halt when confidence score exceeds 0.85
REQ-ORCH-03 Event-Driven Agent Messaging Bus

Introduce a typed event bus for inter-agent communication, replacing the implicit state passing through the scratchpad. This enables agents to subscribe to findings published by peer agents and to trigger downstream workflows reactively.

  • Typed event schema: tool_start | tool_end | finding | confidence_update | plan_revision | error | done
  • Agents publish findings to named channels; downstream agents subscribe selectively
  • External systems (dashboards, alerting, webhooks) subscribe to the bus via SSE or WebSocket
  • Event log persisted to append-only store for full session replay

4.2 Memory & Knowledge Architecture

The single-session scratchpad must be replaced by a tiered memory architecture that provides working memory, episodic recall, and a long-term institutional knowledge base.

REQ-MEM-01 Three-Tier Memory System

Implement a tiered memory model inspired by cognitive architectures, with automatic promotion and eviction policies across tiers.

  • Working Memory — in-context scratchpad for the active query (current behavior, retained as Tier 0)
  • Episodic Memory — vector store of past research sessions, searchable by semantic similarity; allows retrieval of "what did we conclude about NVDA last quarter"
  • Semantic Memory — structured knowledge graph of entities (companies, sectors, executives, macro variables) with time-stamped fact assertions and confidence scores
  • Memory manager with configurable eviction (LRU, time-decay, explicit invalidation)
  • Cross-session context injection: relevant memories automatically injected into new query context
REQ-MEM-02 Institutional Knowledge Base

Allow teams to build a shared, curated knowledge base of domain facts, company profiles, model assumptions, and approved data sources that all agents in the organization inherit.

  • Admins upload structured knowledge documents (10-Ks, earnings call transcripts, internal research) to a private vector store
  • RAG pipeline with hybrid dense+sparse retrieval (BM25 + embeddings) for maximum recall
  • Knowledge staleness tracking: facts flagged for review after configurable TTL
  • Conflict detection: agent flags when new findings contradict existing knowledge base entries

4.3 Tool Ecosystem & MCP Integration

The current tool set must be dramatically expanded and made extensible through a standards-based plugin architecture.

REQ-TOOLS-01 Model Context Protocol (MCP) Native Integration

Adopt MCP as the canonical tool interface standard, enabling any MCP-compatible server to be mounted into Sentinel's tool registry dynamically without code changes.

  • Sentinel acts as an MCP client; tools are discovered via MCP server manifests at startup
  • Hot-reloadable tool registry: new MCP servers can be registered without restarting the agent
  • Tool capability schema validation and sandboxed execution environment per tool
  • Community MCP marketplace for financial tools (SEC EDGAR, Bloomberg API, Refinitiv, Polygon.io, OpenFIGI, etc.)
  • Tool versioning and rollback: agents pin specific tool versions for reproducibility
REQ-TOOLS-02 Expanded Financial Data Tool Surface

Extend the financial data capabilities beyond income/balance/cash-flow statements to cover the full spectrum of data required for institutional-grade research.

  • Options chain data: implied volatility surfaces, open interest, put/call ratios
  • Macro & economic data: FRED, BLS, IMF datasets via standardized tool interface
  • Alternative data: satellite imagery indices, credit card spend aggregates, job postings velocity
  • SEC EDGAR filings: structured 10-K/10-Q/8-K ingestion with entity extraction
  • Earnings call transcripts: NLP-enriched sentiment and management tone scoring
  • Insider trading and institutional 13-F filings
  • ESG data feeds with standardized scoring frameworks
  • Fixed income: yield curves, credit spreads, bond issuance calendars

4.4 Reasoning & Validation Layer

REQ-REASON-01 Independent Validation Agent

Replace self-validation (LLM checking its own work in the same context window) with an independent Validator Agent that receives only the output and the original query — not the intermediate reasoning — and scores it for logical coherence, data consistency, and claim support.

  • Validator runs in a separate context with no access to the primary agent's reasoning chain
  • Structured validation rubric: factual accuracy, temporal consistency, arithmetic verification, source traceability
  • Validator can trigger targeted re-execution of specific tasks when failures are detected
  • Validation scores persisted with each research output for audit purposes
REQ-REASON-02 Chain-of-Thought Audit Trail

Every analytical claim in Sentinel's output must be linked to a verifiable provenance chain: data source, tool call parameters, timestamp, retrieved value, and the reasoning step in which it was used.

  • Output format includes structured citations with source metadata for every factual claim
  • Interactive provenance explorer in the web UI: click any claim to see its data lineage
  • Exportable audit report in PDF/JSON for compliance teams
  • Arithmetic verification: all quantitative calculations include the raw inputs and formula used

Advanced Capability Requirements

5.1 Multi-Agent Collaboration

Complex research tasks demand specialization. A single generalist agent is less effective than a coordinated team of specialists. Sentinel should support configurable multi-agent topologies appropriate for different research workloads.

REQ-MA-01 New
Specialist Agent Pool
Pre-defined specialist agents: Fundamental Analyst, Technical Analyst, Macro Analyst, Sentiment Analyst, Risk Officer, Report Writer. Each has a tailored system prompt, tool subset, and reasoning style.
CriticalArchitecture
REQ-MA-02 New
Orchestrator-Worker Pattern
A top-level Orchestrator decomposes tasks, dispatches to specialist workers, collects results, resolves conflicts between contradictory findings, and synthesizes a unified report.
CriticalArchitecture
REQ-MA-03 New
Red-Team Debate Mode
For high-stakes analyses, spawn a Bull Agent and a Bear Agent who must argue opposite positions, then have a Judge Agent adjudicate and surface the strongest case on each side.
HighReasoning
REQ-MA-04 New
Asynchronous Agent Collaboration
Agents should be able to operate asynchronously — a long-running data retrieval agent should not block a parallel reasoning agent. Results are merged when all dependencies resolve.
HighPerformance

5.2 Adaptive Learning & Personalization

Sentinel should improve its research quality continuously by learning from user feedback, past errors, and the revealed preferences of each user and organization.

REQ-LEARN-01 Feedback-Driven Prompt Optimization
  • Users can rate outputs (thumbs up/down, structured rubric) and provide free-text corrections
  • Feedback is stored and used to fine-tune system prompts via DSPy-style few-shot example injection
  • Error replay: when a query fails or produces a low-rated output, the failure case is saved as a negative example for future prompt optimization
  • A/B evaluation harness: new prompt variants are tested against the eval suite before being promoted to production
REQ-LEARN-02 User & Organization Preference Profiles
  • Per-user profiles: preferred output verbosity, favored analytical frameworks (DCF vs. EV/EBITDA), preferred data sources, watchlist of covered companies
  • Organization profiles: approved data vendors, proprietary benchmark indices, compliance-flagged terminology
  • Profiles automatically influence agent behavior without explicit per-query instructions
  • Profile export and import for analyst onboarding

5.3 Real-Time Streaming & Proactive Alerts

REQ-RT-01 Streaming Research Output

All LLM output and agent events should stream in real-time to the client over SSE or WebSocket, enabling users to see the agent's thinking, tool calls, and partial findings as they emerge rather than waiting for the full response.

  • Server-Sent Events (SSE) endpoint for streaming research progress
  • WebSocket channel for bi-directional interaction (mid-research user interventions)
  • Streaming token-level response rendering in the web UI with typewriter effect
  • Visible tool call trace with live status badges (running, completed, failed)
REQ-RT-02 Proactive Alert Engine

Users should be able to define standing research queries that Sentinel monitors continuously, triggering alerts when conditions change materially.

  • Natural-language alert definitions: "Alert me when AAPL's P/E exceeds its 5-year average by more than 15%"
  • Scheduled research jobs: run research query on a CRON schedule and diff against previous findings
  • Materiality detection: LLM-powered significance scoring to suppress noise and surface only meaningful changes
  • Multi-channel delivery: email, Slack, WhatsApp, webhook, and in-app notifications
  • Alert history and suppression controls to prevent notification fatigue

5.4 Multimodal Research Capabilities

REQ-MM-01 Document & Chart Ingestion

Financial research heavily relies on visual artifacts: charts, tables in PDFs, and presentation decks. Sentinel must be able to ingest and reason over multimodal inputs.

  • PDF ingestion: OCR + layout-aware text extraction from earnings presentations, 10-Ks, pitch decks
  • Chart understanding: vision-model pipeline to extract data series from uploaded chart images
  • Table extraction: structured table parsing from PDFs and HTML pages into queryable data frames
  • User uploads processed into the episodic memory store for future reference
  • Audio/video: transcription of earnings call audio/video recordings fed into the analysis pipeline

Security, Compliance & Trust

Financial services operate under stringent regulatory frameworks. Any platform handling financial research data for institutional clients must implement controls appropriate to that context.

REQ-SEC-01
Authentication & Authorization
OAuth 2.0 / OIDC integration with enterprise IdP (Okta, Azure AD, Google Workspace). Role-based access control with four tiers: Viewer, Analyst, Admin, and Super Admin.
Critical
REQ-SEC-02
API Key Management
Centralized secret vault for all LLM and data API keys. Keys never transmitted to clients. Per-user rate limiting and spend caps per provider to prevent runaway costs.
Critical
REQ-SEC-03
Data Residency & Privacy
Configurable data residency zones. Option to route all LLM calls to on-premise models (Ollama) for air-gapped compliance. PII detection and redaction before external API calls.
High
REQ-SEC-04
Audit Logging
Immutable append-only audit log for all user queries, agent decisions, tool calls, and data accesses. Compliant with SOC 2 Type II and MiFID II record-keeping requirements.
High
REQ-SEC-05
Tool Sandboxing
All MCP tool plugins execute in isolated sandboxes (gVisor or Firecracker microVMs) with network egress whitelisting and filesystem restrictions to prevent supply-chain attacks.
High
REQ-SEC-06
Output Integrity
Research outputs are cryptographically signed with the agent version, model hashes, and tool versions used. Signatures enable verification that outputs have not been tampered with post-generation.
Medium
Compliance Note

Financial research outputs must include prominent disclaimers that they are generated by an AI system and do not constitute investment advice. Sentinel should include configurable disclaimer injection at the organizational level, ensuring compliance with SEC, FCA, and FINRA disclosure requirements.

Observability & Evaluation Framework

Sentinel's existing LangSmith integration provides a solid evaluation foundation. This section specifies the expansions needed to achieve production-grade observability and continuous quality improvement.

REQ-OBS-01 OpenTelemetry Instrumentation
  • Distributed traces for every research query: spans for planning, each tool call, validation, and synthesis
  • Custom span attributes: model, prompt_tokens, completion_tokens, tool_name, confidence_score
  • Compatible with Grafana, Datadog, New Relic, Jaeger, and Honeycomb exporters
  • P50/P95/P99 latency dashboards per tool, per LLM provider, and per query type
REQ-OBS-02 Expanded Evaluation Suite
  • Financial accuracy benchmarks: FinanceBench, CFA Institute sample questions, proprietary ground-truth dataset
  • Regression testing: every code change triggers eval suite; PRs blocked if accuracy drops more than 2%
  • Shadow mode: run new model versions or prompt variants in parallel with production, compare outputs without serving to users
  • Human-in-the-loop evaluation campaigns: periodic expert analyst reviews of sampled outputs with structured rubrics
  • Hallucination rate tracking: automatic fact-checking of all numerical claims against source data
REQ-OBS-03 Cost & Performance Analytics
  • Per-query cost breakdown: LLM tokens (by provider and model), tool API calls, compute time
  • Cost attribution by user, team, and query type for organizational chargeback
  • Efficiency recommendations: flag queries where a cheaper model would likely achieve comparable quality
  • Trend dashboards: cost-per-insight over time as a measure of platform efficiency improvement

Implementation Roadmap

Phase 1 Months 1–3

Foundation — Core Infrastructure Overhaul

Establish the foundational platform capabilities that everything else depends on. No user-facing features are blocked on these but they unlock all subsequent phases.

  • PostgreSQL-backed session persistence replacing JSONL scratchpad files
  • REST API server with OpenAPI spec (Hono or Elysia on Bun)
  • OAuth 2.0 authentication and basic RBAC (Analyst / Admin)
  • OpenTelemetry instrumentation across agent loop and all tools
  • Docker image + one-click Railway/Fly.io deployment
  • MCP client integration with hot-reloadable tool registry
  • Episodic memory store (pgvector) with semantic session search
Phase 2 Months 4–6

Intelligence — Multi-Agent & Advanced Reasoning

Deploy the multi-agent orchestration system and advanced reasoning capabilities that differentiate Sentinel from simpler agent frameworks.

  • Planner Agent with formal ResearchPlan DAG schema
  • Specialist Agent pool: Fundamental, Macro, Technical, Sentiment, Risk
  • Independent Validator Agent with structured validation rubric
  • Chain-of-thought provenance system with source citations
  • Multi-dimensional budget controller (tokens, cost, time, confidence)
  • Expanded financial tool surface: SEC EDGAR, options, macro data, earnings transcripts
  • Red-Team Debate mode (Bull/Bear/Judge topology)
Phase 3 Months 7–9

Experience — Interfaces & Real-Time Capabilities

Build the user-facing surfaces that make Sentinel's intelligence accessible to analysts without requiring CLI proficiency.

  • React web application with streaming research interface
  • Task graph DAG visualization with live status overlay
  • Research history, knowledge base browser, and annotation tools
  • Proactive Alert Engine with natural-language alert definition
  • Multi-channel notification delivery (Slack, email, webhook)
  • PDF/document ingestion pipeline for uploaded research materials
  • TypeScript and Python SDK packages with comprehensive documentation
Phase 4 Months 10–12

Scale — Enterprise Readiness & Learning Systems

Add the enterprise-grade features required for institutional deployment and establish the learning loops that enable continuous quality improvement.

  • Institutional Knowledge Base with admin-curated RAG pipeline
  • Feedback-driven prompt optimization (DSPy-style few-shot injection)
  • User and organization preference profiles
  • Full multi-tenancy with organizational workspaces and team collaboration
  • MCP marketplace for community and vendor tool plugins
  • SOC 2-aligned audit logging and compliance controls
  • Kubernetes Helm chart + enterprise support tier
